# ChargeFlow Agent 面试问答清单

> 针对 AI 产品经理 / Agent 技术岗位，基于 ChargeFlow Agent 项目准备的高频面试问答。

---

## 一、产品设计类

### Q1：ChargeFlow Agent 解决的核心问题是什么？和市面上的"找桩 App"有什么区别？

**A：** 市面上的找桩 App 是被动工具——用户电量低了，手动打开 App 搜索充电站。ChargeFlow Agent 是主动决策 Agent，核心区别在于：

1. **任务感知**：它不只看电量，还看你当前在做什么（导航中？空闲？）
2. **未来预判**：结合日历行程计算"最晚补能时间"，而不是等到电量见底才提醒
3. **跨时段续接**：上次没充电就下车了，下次上车自动恢复上次的补能建议
4. **多因素决策**：综合距离、充电速度、等待时间、用户偏好做推荐，而不只是"最近的桩"

一句话总结：从"用户找桩"变成"Agent 帮你管能源"。

---

### Q2：你是怎么定义这四个场景的？为什么是这四个？

**A：** 我从电动车车主的真实补能决策链路出发，按"当前任务状态"做了状态空间划分：

| 当前状态 | 是否有未来行程 | 场景 |
|---------|-------------|------|
| 空闲（无导航） | 无 | A — 主动补能 |
| 忙碌（导航中） | — | B — 保障当前行程 |
| 空闲（无导航） | 有 | C — 预判未来出行 |
| 任何状态 | 有历史未完成任务 | D — 跨会话续接 |

这四个场景覆盖了从"无事可做"到"正在赶路"到"有未来计划"再到"跨时段记忆"的完整状态空间。我排除了一些低频场景（比如正在充电中、电池故障等），因为作为 portfolio 项目优先展示决策逻辑的完整性。

---

### Q3：场景 B "导航途中" 的决策逻辑是怎么设计的？为什么不直接中断导航去充电？

**A：** 这是一个用户体验判断。对于正在赶路的司机来说，被 Agent 强制中断导航去充电是很糟糕的体验。所以设计了分级策略：

- **续航够用**：不中断，但告知"最晚在返程前/到达后需要充电"，锁定补能截止点
- **续航不够到目的地**：这才触发紧急改道，因为这时不充电会抛锚

这个分级体现了一个产品原则：**Agent 的建议应该尊重用户当前的任务优先级**，而不是一味追求"最优充电时机"。

---

### Q4：场景 C 的"最晚补能时间"是怎么算的？

**A：** 计算公式：

```
最晚补能时间 = 日程开始时间 - 预计行驶时间 - 充电耗时 - 安全缓冲
```

需要的数据：
- 日程开始时间 → `get_calendar_events`
- 目的地距离 → 日历事件中的 `distance_km`
- 当前续航 → `get_vehicle_status`
- 充电耗时 → `search_nearby_stations` 返回的 `estimatedChargeTime_min`
- 安全缓冲 → 硬编码 15~30 分钟

这个计算由 LLM 在 Prompt 的场景规则指导下完成，而不是写在后端代码里。这是 Agent 架构的优势——决策逻辑在 Prompt 中，而不是 if-else。

---

### Q5：你怎么衡量这个 Agent 的效果？如果要上线，KPI 怎么定？

**A：**

| 指标 | 定义 | 目标 |
|------|------|------|
| 场景识别准确率 | Agent 正确识别当前属于哪个场景 | > 90% |
| 推荐接受率 | 用户接受充电建议并执行的比例 | > 40% |
| 任务续接率 | 跨会话的 pending task 被用户最终执行的比例 | > 25% |
| 决策响应时间 | 从用户输入到 Agent 给出完整建议的耗时 | < 5s |
| 补能及时率 | 用户因 Agent 建议在行程前完成充电的比例 | > 70% |

面试亮点：不要只说"准确率"，要说**用户行为指标**（接受率、续接率）和**业务指标**（补能及时率）。

---

## 二、Agent 技术类

### Q6：你的 Agent 用了什么架构模式？和 ReAct / Plan-and-Execute 有什么区别？

**A：** ChargeFlow Agent 使用的是 **Tool-augmented LLM + Scenario-guided Prompt** 架构：

- **不是 ReAct**：ReAct 是"思考-行动-观察"的通用循环，每步由 LLM 自己决定下一步做什么。我的 Agent 有明确的场景规则，LLM 不需要从零推理该调什么工具
- **不是 Plan-and-Execute**：没有显式的"先生成计划再执行"步骤，而是在 System Prompt 中内置了场景决策树，LLM 一次性完成判断和工具调用
- **更像 Rule-guided Tool Orchestration**：场景规则是人工设计的，工具选择和组合是 LLM 完成的

这种架构的优势是**可控性强**——产品经理可以通过修改场景规则直接影响 Agent 行为，不需要重新训练模型。

---

### Q7：Prompt 为什么分五层？每层的作用是什么？

**A：**

```
角色层 → 场景规则层 → 工具层 → 记忆层 → 约束层
```

1. **角色层**：定义 Agent 身份（座舱能源管家），设定"主动、简洁、安全优先"的行为基调
2. **场景规则层**：四大场景的触发条件和行动指南——这是核心创新点，让 LLM 有明确的决策框架而不是自由发挥
3. **工具层**：说明 5 个工具的用途和优先级（先查车辆状态再做推荐），防止 LLM 跳过数据收集直接给建议
4. **记忆层**：注入用户的充电偏好和驾驶习惯，实现个性化
5. **约束层**：输出格式（简洁、中文为主）、异常处理（用户拒绝时存为 pending task）

面试亮点：强调"场景规则层"是这个 Prompt 设计和通用助手最大的区别——它把产品逻辑编码进了 Prompt。

---

### Q8：为什么用 Function Calling 而不是让 LLM 直接生成答案？

**A：** 三个原因：

1. **防幻觉**：电量 18% 还是 80%、附近有没有空闲桩，这些必须查真实数据，不能让 LLM 编。Prompt 里明确写了 "NEVER guess data — always use tools"
2. **可追溯**：前端有工具调用链路可视化，面试官（或未来的用户）可以看到 Agent 的推理过程——"它先查了车辆状态，再搜了附近站点，最后基于这些数据给出建议"
3. **可扩展**：工具是 registry 模式注册的，后续加"实时路况查询""充电费用计算"只需要加工具定义和实现，不需要改 Prompt 核心逻辑

---

### Q9：工具调用的编排顺序是谁决定的？LLM 还是你的代码？

**A：** **LLM 决定调用顺序，Prompt 提供优先级指导。**

- Prompt 中写了"Always call get_vehicle_status first"，这是一个软约束
- LLM 根据用户意图和场景规则决定后续调哪些工具
- 代码层面是顺序执行：收到 tool_use block → 执行 → 把 tool_result 回传 → LLM 决定是否需要调更多工具

当前实现是单轮工具调用（一次 LLM 调用中的所有 tool_use 都执行完再回传）。如果需要多轮工具调用（一个工具的结果决定下一个工具的参数），需要循环执行直到 LLM 返回 `end_turn`。

---

### Q10：跨会话记忆是怎么实现的？和 RAG 有什么区别？

**A：** 我实现了两套持久化：

**1. 驾驶偏好记忆（memory.json）**
- 用启发式规则从对话中提取偏好（如"我偏好特斯拉超充"）
- 每次对话开始时注入 System Prompt
- 类似 RAG 的"检索+注入"，但数据源是用户自己的对话历史而不是外部文档

**2. 未完成任务（pending_tasks.json）**
- 充电建议被拒绝时存为 pending task
- 下次对话时通过 `get_pending_charge_tasks` 工具读取
- 这不是记忆，是**任务状态管理**

和 RAG 的区别：
- RAG 是从大规模文档中检索相关片段，用向量相似度匹配
- 我的记忆系统是小规模、结构化的 JSON，用规则提取，全量注入
- 如果记忆条目变多（比如 100+ 条），就需要升级为 embedding + 向量检索的 RAG 方案

---

### Q11：mock mode 是怎么设计的？为什么需要它？

**A：** mock mode 在没有 Anthropic API Key 时自动启用，用关键词匹配模拟 Agent 的四大场景响应。

设计原因：
1. **Portfolio 可演示性**：面试官 clone 项目后不需要付费 API 就能看到完整的交互流程
2. **开发效率**：开发前端和调试工具调用链路时不消耗 API 额度
3. **架构验证**：mock 的响应格式和真实 API 完全一致（message + toolCalls + memory），证明前后端协议是正确的

代码中通过 `!anthropic || process.env.MOCK_MODE === 'true'` 判断，降级逻辑在 `createMockResponse` 函数中。

## 三、工程决策类

### Q12：为什么选 React + Express 而不是 Next.js 或纯前端？

**A：**
- **Express 独立后端**：Agent 的工具执行、记忆持久化、LLM 调用都需要服务端逻辑，独立后端更清晰
- **React SPA**：座舱场景下 UI 是持续在线的单页应用，不需要 SSR
- **没用 Next.js**：这个项目的重点是 Agent 架构而不是前端工程，Next.js 会引入不必要的复杂度

---

### Q13：如果要让这个 Agent 支持多用户/多车辆，需要改什么？

**A：** 当前是单用户 JSON 文件存储，需要改三层：

1. **存储层**：JSON → 数据库（PostgreSQL / MongoDB），按 userId + vehicleId 隔离
2. **认证层**：加 JWT / OAuth，API 路由加鉴权中间件
3. **Agent 层**：System Prompt 中的记忆注入需要按用户加载，工具查询需要带 vehicleId 参数

改动量不大，因为当前架构已经把数据访问封装在 `readJson/writeJson` 里，替换存储层不影响上层逻辑。

---

### Q14：你怎么处理 LLM 的延迟问题？座舱场景对实时性要求很高。

**A：** 目前的实现是同步请求-响应，LLM 调用 + 工具执行大约 3-5 秒。优化方向：

1. **Streaming**：用 Anthropic 的 streaming API，让 UI 逐字显示，体感更快
2. **预计算**：车机启动时后台自动触发一次场景 A/D 的检查，结果缓存，用户开口前就准备好了
3. **工具并行**：当前是顺序执行工具，如果 LLM 同时调了 vehicle_status 和 calendar，可以 `Promise.all` 并行执行
4. **小模型前置**：用轻量模型做意图分类，只有复杂场景才调大模型

---

### Q15：如果 LLM 给出了错误的充电建议（比如推荐了一个已满的站），怎么处理？

**A：** 分三层防御：

1. **数据层**：工具返回的站点数据包含 `availablePorts` 字段，Prompt 指导 LLM 优先推荐有空闲桩的站点
2. **Prompt 层**：约束层要求 LLM 解释推荐原因，如果它推荐了满桩的站点，原因中会暴露矛盾
3. **产品层**：如果要上线，需要加一个确认环节——用户点击"导航前往"时再次实时查询站点状态，发现已满则更新推荐

关键认知：**LLM Agent 的输出是建议而不是执行，执行前有确认环节就能兜底。**

---

## 四、高阶 / 开放性问题

### Q16：你觉得 Agent 和传统的规则引擎在这个场景下各有什么优劣？

**A：**

| 维度 | 规则引擎 | LLM Agent |
|------|---------|-----------|
| 确定性 | 高，逻辑硬编码 | 中，取决于 Prompt 质量 |
| 灵活性 | 低，新场景需要写新规则 | 高，自然语言描述即可 |
| 可解释性 | 高，每条规则可追溯 | 中，需要工具调用链路辅助 |
| 自然语言交互 | 不支持 | 天然支持 |
| 开发成本 | 场景多时爆炸式增长 | 相对稳定 |

我的选择：**用 Prompt 中的场景规则代替硬编码规则引擎，用 LLM 做规则的解释和执行**。这样既有规则引擎的可控性，又有 LLM 的灵活性。

---

### Q17：如果让你从零设计一个真实的车载充电 Agent，你会怎么做？

**A：** 分三个阶段：

**Phase 1 — MVP（当前项目的延伸）**
- 接入高德地图 API 获取真实路径和充电站 POI
- 接入车辆 OBD 数据获取真实 SOC
- 单车单用户，验证四大场景的用户接受率

**Phase 2 — 规模化**
- 接入多家充电网络的实时数据（桩位状态、价格、排队）
- 加入预约充电功能
- 多车型适配（不同车型充电曲线不同）
- A/B 测试推荐策略

**Phase 3 — 智能化**
- 学习用户的充电习惯（什么时间充电、偏好什么价位）
- 预测性补能（根据日常通勤模式提前建议）
- 和车机的语音助手集成，支持语音交互
- 群体智能：基于同路线其他车主的充电行为优化推荐

---

### Q18：你这个项目中用到了哪些 Prompt Engineering 技巧？

**A：**

1. **分层结构（Layered Prompting）**：角色 → 场景 → 工具 → 记忆 → 约束，每层解决不同问题
2. **场景规则注入（Scenario Injection）**：把产品逻辑以自然语言形式写入 Prompt，而不是代码
3. **工具优先级引导**：明确写"Always call get_vehicle_status first"，控制工具调用顺序
4. **反幻觉约束**："NEVER guess data — always use tools"
5. **输出格式约束**："Always state current SOC and estimated range"
6. **记忆注入（Memory Injection）**：将持久化的用户偏好以 bullet list 格式注入 Prompt
7. **异常路径设计**：用户拒绝时怎么处理、信息缺失时问什么——这些边界情况都在 Prompt 中覆盖了

---

### Q19：你怎么看 Agent 的"幻觉"问题在座舱场景下的风险？

**A：** 座舱场景幻觉风险极高，因为错误建议可能导致车辆抛锚。我的防御策略：

1. **数据源隔离**：所有关键数据（电量、距离、站点）必须通过工具获取，Prompt 禁止猜测
2. **决策可追溯**：前端展示完整的工具调用链路，用户可以验证 Agent 的推理依据
3. **保守策略**：安全缓冲设计——续航计算时留 15-30 分钟余量，宁可早充不可晚充
4. **人在回路**：Agent 给建议，用户确认执行。不做全自动操作（比如不会自动改导航路线）

核心原则：**在安全关键场景中，Agent 应该是 advisor 而不是 executor。**

---

### Q20：这个项目对你理解 AI Agent 最大的收获是什么？

**A：（参考，请用自己的话改写）**

最大的收获是理解了 **Agent 的核心价值不是调用工具，而是在上下文中做出合理的决策编排**。

一个好的 Agent 产品，不是把 5 个 API 包装成聊天界面，而是要回答：
- 什么时候该主动说话？（场景 A：SOC 低就该提醒）
- 什么时候该克制？（场景 B：导航中不要强制中断）
- 什么时候该记住？（场景 D：未完成的任务要跨会话延续）
- 什么时候该综合判断？（场景 C：当前电量够用但未来不够）

这些决策背后是产品设计，不是工程实现。工程只是让这些决策跑起来的手段。

---

### Q21：这个 Agent 怎么部署到真实车机上？需要集成到座舱中控系统吗？能独立使用吗？

**A：** 当前项目已经是独立可运行的 Web 应用，不需要集成到任何车机系统。真实上车有三条路径，从轻到重：

**路径 1：手机端（最轻量，当前就能做）**

```
司机手机浏览器 / 小程序 / PWA
        │
        ▼
  云端 Express 服务
        │
        ├── Claude API（决策）
        ├── 高德 API（地图）
        └── 充电网络 API（站点数据）
```

- 不碰车机，完全独立
- 车辆 SOC 数据来源：用户手动输入，或通过车厂开放 API 读取（特斯拉/蔚来/小鹏都有）
- 大部分充电 App 就是这个做法

**路径 2：车机 WebView（中等集成，最现实的上车方案）**

```
车机中控屏（内置 WebView 容器）
        │
        ▼
  React UI（打包成 H5 页面）
        │
        ▼
  云端 Agent 服务
```

- 现在主流车机系统（Android Automotive、QNX、鸿蒙座舱）都支持 WebView
- React 应用打包成 H5，通过车机应用商店或 URL 加载
- 车辆数据通过车机的 **JS Bridge** 获取：

```js
// 车机 JS Bridge 获取真实车辆数据（替代当前的 JSON mock）
const soc = await window.CarBridge.getSOC();
const location = await window.CarBridge.getGPS();
const navState = await window.CarBridge.getNavState();
```

- 蔚来、小鹏、理想的车机第三方应用大量采用这种方案
- 对当前项目的改动：加一个 JS Bridge 适配层替代 JSON 文件读取即可

**路径 3：深度集成座舱 OS（最重，需要车厂合作）**

```
座舱域控制器（高通 SA8295P 等）
        │
        ├── 车辆信号总线（CAN / SOME-IP）→ 实时 SOC、GPS、导航
        ├── 座舱 OS → Native App（Kotlin / ArkTS）
        │               └── 调用云端 Agent API
        └── 语音助手集成
                └── 语音 → 文本 → /api/chat → 语音播报
```

- 需要和车厂或 Tier 1 供应商合作
- 重写为 Native App + 对接车辆信号总线

**面试推荐话术：**

> "当前是独立 Web 应用，已经能用。如果要上车，最现实的路径是走车机 WebView——把 React 打包成 H5，车辆数据通过 JS Bridge 从车机 SDK 获取，Agent 决策仍然在云端完成。这也是蔚来/小鹏车机上大部分第三方应用的集成方式。"

---

### Q22：如果车机没有网络（比如地下车库），Agent 怎么工作？

**A：** 这是一个很好的边界场景。分两层处理：

1. **离线降级**：本地缓存上一次的站点搜索结果和 pending tasks，断网时展示缓存数据并标注"数据可能不是最新"
2. **LLM 降级**：当前项目的 mock mode 本身就是一种离线方案——不依赖 Claude API 也能给出基于规则的基础建议
3. **重连恢复**：网络恢复后自动刷新车辆状态和站点数据，更新推荐

产品原则：**离线时给旧建议 > 不给建议。用户看到"上次推荐的充电站在 0.8km 外"比看到"网络错误"有用得多。**

## 五、用户为中心的设计思维

### Q23：这个项目是怎么体现"以用户为中心"的设计思想的？

**A：** 不是加了个聊天框就叫"以用户为中心"。这个项目在五个层面体现了这个思想：

#### 1. 场景来自真实用户痛点，不是技术炫技

我没有先想"LLM 能做什么"，而是先想**电动车车主的焦虑是什么**：

| 用户真实痛点 | 对应场景 |
|-------------|---------|
| 电量低了不知道去哪充 | 场景 A：主动推荐 |
| 赶路时怕半路没电 | 场景 B：保障当前行程 |
| 明天要跑长途，不确定要不要提前充 | 场景 C：预判未来出行 |
| 上次提醒了没充，这次又忘了 | 场景 D：跨会话续接 |

四个场景不是"我觉得用户需要"，而是从真实电车车主论坛（蔚来 NIO App、小鹏社区、懂车帝）的高频抱怨中提炼的。

#### 2. Agent 尊重用户当前任务，不强制中断

这是最关键的设计决策。场景 B 的设计：

```
❌ 错误做法：电量低 → 无条件弹窗"请立即充电" → 中断导航
✅ 我的做法：电量低 + 正在导航 → 判断够不够到目的地 → 够用就不打扰，只告知截止点
```

**设计原则：Agent 不应该比用户更"着急"。** 用户正在赶路开会，被 Agent 强制改道去充电是灾难性的体验。好的 Agent 是给信息让用户自己判断，而不是替用户做决定。

#### 3. 推荐逻辑考虑用户偏好，不只是"最近的桩"

找桩 App 的默认排序是距离。但真实用户有偏好：

- 有人只认特斯拉超充（兼容性、充电速度）
- 有人在意价格（国网便宜但慢）
- 有人讨厌排队（宁可多开 3km 去没人的站）

所以 Agent 有记忆系统，会记住：
```
- [preference] User prefers Tesla Supercharger and NIO Power stations.
- [preference] User avoids slow charging stations under 100kW.
```

推荐时结合偏好排序，而不是一刀切。**个性化不是加分项，是尊重用户差异的基本功。**

#### 4. 信息架构为驾驶场景优化，不是"塞满功能"

驾驶中用户注意力极度有限。UI 设计上做了三个取舍：

| 设计决策 | 用户为中心的理由 |
|---------|---------------|
| 车辆状态仪表盘放最顶部 | 司机第一眼就能看到电量和续航，不需要对话就能获取关键信息 |
| Agent 回复要求简洁 | Prompt 约束层写了"drivers read at a glance"，回复不超过 3-4 行 |
| Demo 场景面板用 A/B/C/D 标签 | 降低用户的认知负荷，一眼知道 Agent 能干什么 |
| 工具调用链路默认展示 | 让用户知道"Agent 的建议是基于什么数据的"，建立信任 |

**反面教材：** 很多 AI 产品把"Agent 能力展示"和"用户实际需求"混在一起，界面塞满技术细节。座舱场景下，用户要的是"该不该充、去哪充、什么时候充"，不是"我调用了 5 个 API"。

#### 5. 失败路径也为用户设计，不是报错了事

大部分 Agent demo 只展示 happy path。这个项目覆盖了用户会遇到的失败场景：

| 失败场景 | 传统做法 | 我的做法 |
|---------|---------|---------|
| 用户拒绝充电建议 | 丢弃，下次再算 | 存为 pending task，下次上车自动恢复 |
| 没有 API Key | 报错 | mock mode 降级，照样能跑完整流程 |
| 推荐的站全满 | 只显示"无可用站点" | 扩大搜索半径，或建议等待时间最短的站 |
| 断网 | "网络错误" | 展示缓存数据 + 标注"数据可能不是最新" |

**设计原则：用户遇到问题时，Agent 应该给出"下一步能做什么"，而不只是说"出错了"。**

#### 面试总结话术

> "以用户为中心不是一句口号，在这个项目中它体现在五个具体决策上：场景从真实痛点出发而不是技术能力出发；Agent 尊重用户当前任务不强制中断；推荐考虑个人偏好而不只是最短距离；UI 信息层级为驾驶注意力优化；失败路径也给出下一步行动而不是报错了事。每一个决策都可以指着代码讲清楚为什么这么做。"

### Q24：你怎么验证这个 Agent 的推荐"真的对用户有用"而不是"技术上能跑通"？

**A：** 这是区分"技术 demo"和"产品 demo"的关键问题。我会从三个维度验证：

**1. 场景覆盖度测试（定性）**

设计 10-15 条用户故事，覆盖四大场景的典型和边界情况：

```
✅ 电量 18%，无目的地 → 应推荐最优站
✅ 电量 50%，正导航去 3km 外的会议 → 不应中断
✅ 电量 50%，后天要去 35km 外的机场 → 应提前提醒
✅ 电量 80%，无目的地 → 不应打扰用户
✅ 上次推荐被拒，这次上车 → 应恢复旧任务
❌ 推荐了一个距离最近但全满的站 → 这是 bad case
```

每条人工走一遍，验证 Agent 响应是否符合预期。

**2. 推荐质量评估（定量）**

如果有真实用户，核心看两个指标：
- **推荐接受率**：用户看到建议后真的去充电了 → 说明建议有用
- **推荐时机满意度**：用户没有觉得"太早/太晚/太打扰" → 说明场景判断准

如果是 portfolio 阶段，可以用 **A/B 对比法** 自测：对同一个场景，比较"只按距离推荐"和"综合偏好+场景推荐"的结果差异。

**3. 用户旅程走查（体验）**

把自己当成一个赶着去开会的车主，从上车到停车走完整流程：
- 上车时 Agent 说了什么？是有用的信息还是废话？
- 开到一半电量告警，Agent 的反应让我安心还是焦虑？
- 到了目的地 Agent 有没有提醒我充电再回家？
- 第二天上车，Agent 还记得昨天的事吗？

**一句话：技术上"能跑通"只是 60 分。用户觉得"这个建议来得正好"才是 90 分。**

### Q25：如果用户反馈"Agent 太啰嗦/太烦了"，你怎么调整？

**A：** 这是 Agent 产品最常见的负反馈。本质问题是**触发频率和信息密度没有匹配用户预期**。调整分三层：

**1. 触发阈值调优**

```
当前：SOC < 20% 就触发提醒
调整：根据用户历史行为动态调整
  - 这个用户经常开到 10% 才充电 → 阈值降到 15%
  - 这个用户从不低于 40% → 阈值升到 40%
```

记忆系统已经能记住用户偏好，只需要加一条：`"User usually charges when SOC drops to around 10%."`，Prompt 中的场景规则就会据此调整触发时机。

**2. 信息分级**

不是每次都需要完整推荐。设计三个信息级别：

| 级别 | 场景 | Agent 输出 |
|------|------|-----------|
| 轻提示 | SOC 偏低但不紧急 | "当前 35%，续航 120km。" （一句话，不推荐站点）|
| 标准建议 | SOC 低且有行程影响 | "当前 18%，建议去 XX 站充电，预计 35 分钟充至 80%。" |
| 紧急告警 | 续航不足以到目的地 | "续航不足！最近充电站在 0.8km 外，建议立即前往。" |

**3. 用户可控**

加一个偏好设置："提醒频率：积极 / 标准 / 安静"，存入记忆系统。这让用户自己决定 Agent 的"话多程度"，而不是产品经理替所有人做决定。

**核心认识：好的 Agent 不是说得越多越好，而是在正确的时间说正确的话。沉默有时候也是一种好的用户体验。**
