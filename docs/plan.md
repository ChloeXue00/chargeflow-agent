# ChargeFlow Agent 构建计划

> 本文档记录了从产品构思到 Agent 实现的完整工作流。
> 采用 "Structured Prompt → Plan → Agent Generation" 方法论，而非传统的逐行编码。

---

## 一、方法论说明

### 我用了什么方法？

本项目采用 **AI-assisted structured development**（结构化 AI 辅助开发），工作流如下：

```
Step 1: 场景洞察（人）
    ↓
Step 2: 产品规则设计（人）
    ↓
Step 3: 结构化 Prompt / Plan（人 → AI）
    ↓
Step 4: Agent 代码生成（AI）
    ↓
Step 5: 验证与迭代（人 + AI）
```

**和纯 vibe coding 的区别：**

| 维度 | 纯 Vibe Coding | 本项目的做法 |
|------|----------------|-------------|
| 输入 | "帮我做一个充电桩 App" | 四大场景 × 触发条件 × 系统动作 × 设计原则 |
| 人的角色 | 提需求 → 等结果 | 定义场景规则 → 审查架构 → 迭代优化 |
| AI 的角色 | 全权设计+实现 | 在人给定的框架内实现 |
| 可控性 | 低，结果不可预测 | 高，每个场景的行为有明确预期 |
| 可复现性 | 低 | 高，同样的 plan 给另一个 AI 也能生成类似系统 |

**一句话总结：vibe coding 是"让 AI 自由发挥"，我的做法是"人定规则，AI 写代码"。**

## 二、Step 1 — 场景洞察

### 输入：一个模糊的产品直觉

> "不只是电量低了再找桩，而是一个能主动管理补能的智能座舱 Agent。"

### 输出：四大场景的自然语言描述

通过分析电动车车主的真实补能决策链路，按"当前任务状态 × 未来行程"做状态空间划分：

| # | 场景 | 一句话逻辑 |
|---|------|-----------|
| A | 无目的地、无紧急行程 | 没有行程约束时，优先立即补能 |
| B | 正在前往目的地途中 | 导航途中优先保障当前任务，但提前锁定补能截止点 |
| C | 无导航，但有后续日程 | 不只看当前电量，而是看是否影响后续履约 |
| D | 上次未执行补能，本次上车 | Agent 具备跨时段记忆与续接能力 |

### 这一步的核心价值

**场景定义是产品经理的核心输出，不是 AI 能替你做的。** AI 可以帮你写代码、写 Prompt、搭架构，但"这个产品应该覆盖哪些场景、每个场景的优先级是什么"——这是人的判断。

## 三、Step 2 — 产品规则设计

### 输入：四大场景

### 输出：每个场景的结构化规则

为每个场景定义三要素：**触发条件 → 系统动作 → 设计原则**

```yaml
Scenario A:
  trigger:
    - SOC < 20%
    - 无导航目的地
    - 短期内无高优日程
  action:
    - 自动检索附近充电站
    - 按距离、等待时间、可用桩数、充电效率排序
    - 生成最优补能方案
    - 用户可一键导航
  principle: 没有行程约束时，优先立即补能

Scenario B:
  trigger:
    - 正在导航中
    - 剩余电量可能影响到达或返程
  action:
    - 判断续航 vs 剩余路程
    - 够用 → 不中断，提示最晚补能截止点
    - 不够 → 重规划最近补能点
  principle: 导航途中优先保障当前任务，但提前锁定补能截止点

Scenario C:
  trigger:
    - 当前无导航
    - 日历有后续需驾车的日程
    - 当前电量不足以支持出行链路
  action:
    - 基于日程时间、距离、充电耗时计算"最晚补能时间"
    - 建议空闲时段充电
    - 保留为待执行任务
  principle: 不只看当前电量，看是否影响后续履约

Scenario D:
  trigger:
    - 上一次有未完成的补能建议
    - 本次车机启动
  action:
    - 读取未完成任务
    - 重新评估当前条件
    - 更新推荐
  principle: 跨时段记忆与续接
```

### 这一步的核心价值

这个 YAML 就是给 AI 的"产品规格书"。它足够具体到 AI 可以直接实现，但又足够抽象到不限制技术选型。

## 四、Step 3 — 结构化 Prompt（给 AI 的指令）

### 输入：场景规则

### 输出：一段结构化的 generation prompt

实际上我给 AI（Claude Code）的指令大致是这样的：

```markdown
# 任务
把这个项目从"个人助手 Agent"改造成"智能座舱补能决策 Agent"。

# 核心场景规则
[粘贴 Step 2 的四大场景规则]

# 需要的工具（Function Calling）
1. get_vehicle_status — 获取 SOC、续航、位置、导航状态
2. search_nearby_stations — 搜索附近充电站，支持按距离/功率/价格/空闲排序
3. get_calendar_events — 查询日历行程，评估补能紧迫性
4. get_pending_charge_tasks — 读取未完成的充电任务
5. create_charge_plan — 创建充电计划并持久化

# 系统 Prompt 设计要求
- 五层结构：角色 → 场景规则 → 工具 → 记忆 → 约束
- 场景规则直接写进 Prompt，让 LLM 根据车辆状态自动选择场景
- 禁止 LLM 猜测数据，必须通过工具获取

# 数据模型
- vehicle_state.json — 模拟一辆 SOC 18%、续航 62km、在上海黄浦区的电车
- charging_stations.json — 5 个上海充电站（特斯拉/蔚来/国网/小鹏）
- calendar.json — 包含驾车场景的日程（客户会议、机场接机）
- pending_tasks.json — 一条上次未执行的充电建议
- memory.json — 用户偏好（偏好特斯拉超充、避免慢充）

# UI 要求
- 顶部：车辆状态仪表盘（SOC、续航、导航状态、位置）
- 中间：对话窗口 + 工具调用链路
- 右侧：记忆面板 + 四大场景 Demo 入口

# Mock mode
- 无 API Key 时自动降级
- 覆盖四大场景的关键词触发
```

### Prompt 写法要点

| 要点 | 为什么 |
|------|--------|
| **先定义场景规则，再说技术实现** | AI 需要知道"做什么"比"怎么做"更重要 |
| **列出所有工具和它们的用途** | 防止 AI 自己发明奇怪的工具 |
| **给出具体的数据样例** | AI 生成的 mock 数据才有真实感 |
| **说明约束和边界** | "禁止 LLM 猜测数据"这类负面约束很关键 |
| **描述 UI 布局** | 不说的话 AI 会用它自己的审美 |

## 五、Step 4 — AI 生成 + 人工审查

### AI 生成清单

| 文件 | AI 生成 | 人审查重点 |
|------|---------|-----------|
| server/services/tools.js | 5 个工具实现 + Anthropic schema | 参数校验是否完整、排序逻辑是否合理 |
| server/services/llm.js | 五层 System Prompt + mock mode | 场景规则是否准确还原了 Step 2 的设计 |
| server/services/memory.js | 偏好提取规则 | 正则是否覆盖中英文 |
| server/data/*.json | 5 个 mock 数据文件 | 数据是否自洽（距离、SOC、续航之间的关系） |
| client/src/ | 座舱 UI 组件 | 信息层级是否符合驾驶场景 |
| docs/ | PRD、架构、Prompt 设计 | 文档是否准确反映实现 |

### 人工审查要点

生成后不是直接用，而是检查：

1. **场景覆盖**：每个场景都能通过对话触发吗？mock mode 覆盖了四个场景吗？
2. **数据自洽**：SOC 18% 续航 62km 合理吗？（75kWh × 18% ≈ 13.5kWh，按 5km/kWh 约 67km ✓）
3. **Prompt 忠实度**：System Prompt 中的场景规则是否准确还原了 YAML 规格？
4. **工具链路**：场景 C 需要同时调 vehicle_status + calendar，代码支持多工具链路吗？

## 六、Step 5 — 迭代优化记录

| 迭代 | 发现的问题 | 调整 |
|------|-----------|------|
| v1 | 项目还是"个人助手"定位 | 全量改造为座舱补能 Agent |

## 七、可复用的工作流模板

如果你要用同样的方法做另一个 Agent 项目，按这个顺序：

```
1. 定义用户是谁、核心痛点是什么（1 段话）
2. 划分场景状态空间（表格）
3. 每个场景写 trigger → action → principle（YAML）
4. 定义需要的工具列表和数据模型（列表）
5. 写结构化 Prompt 给 AI（Markdown）
6. AI 生成 → 人工审查 → 迭代
```

**这个流程的核心认知：人负责"做什么"和"为什么"，AI 负责"怎么做"。Plan 是两者之间的接口契约。**
